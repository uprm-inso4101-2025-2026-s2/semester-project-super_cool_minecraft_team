=== 1.3 Current Situation

[.line-through.red]#Minecraft mods rely on other mods, loaders, libraries, and specific Minecraft versions; these are the mod dependencies.
If these dependencies are not met, the game will crash or fail to launch.
This leads to user frustration and wasted time.
These dependencies are not easy to access; while it would be better to have them on the main download page, they are usually provided as plain text in metadata or in dedicated wikis that less-experienced users may not find.#

[.line-through.red]#Minecraft mods frequently depend on other mods, libraries, specific mod loaders, and particular Minecraft versions in order to function correctly. These requirements form a set of dependencies that must be satisfied before the game can run successfully.#

[.line-through.red]#If these dependencies are missing or incompatible, Minecraft may fail to launch or crash during runtime. This creates frustration for players and often leads to wasted time troubleshooting mod configurations.#

[.line-through.red]#One of the primary difficulties is that dependency information is not always presented in an accessible format. In many cases, the required dependencies are only listed in technical metadata files or scattered across community-maintained documentation and wikis. As a result, players who are unfamiliar with modding terminology or metadata structures often struggle to understand which additional mods are required or why conflicts occur.#

[.line-through.red]#This lack of clear and centralized dependency information makes modpack configuration unnecessarily complex, particularly for less experienced players.#

Minecraft modding has become increasingly complex because many mods depend on specific loaders, libraries, and Minecraft versions in order to work correctly.

For example, a player may install Sodium for performance improvements on Fabric 1.20.1, only to discover that another installed mod requires Forge or NeoForge instead of Fabric. In other situations, mods such as Create or Farmer’s Delight may require additional libraries like Flywheel or Architectury API. If these dependencies are missing or incompatible, Minecraft may crash during startup, freeze while loading, or display confusing error messages that many players do not understand.

Another common problem occurs when players mix mods designed for different Minecraft versions. A mod built for Minecraft 1.19.2 may appear compatible at first but fail completely when used in 1.20.1. Modpacks with dozens or even hundreds of mods make these problems significantly harder to identify manually.

Although the required information usually exists somewhere, it is often scattered across CurseForge pages, GitHub repositories, Discord servers, wiki pages, or metadata files such as fabric.mod.json and mods.toml. Less experienced players frequently struggle to determine which dependency is missing, which mod is causing a conflict, or whether the issue is related to the mod loader itself.

As a result, players often spend large amounts of time troubleshooting crashes, removing mods one by one, searching through logs, and reinstalling modpacks instead of actually playing the game.

=== 1.4 Needs and Ideas

[.line-through.red]#We identify that many Minecraft mods rely on other mods, libraries, specific loaders, and specific Minecraft versions in order to function correctly.
When dependencies are missing, incompatible, or incorrectly ordered, the game often fails to launch or crashes during runtime.#

Minecraft mods often depend on other mods, libraries, mod loaders, and specific Minecraft versions in order to function correctly. When these dependencies are missing, incompatible, or loaded in the wrong order, the game may fail to launch or crash during runtime.

[.line-through.red]#There is a significant lack of accessibility in dependency documentation.
Currently, users are forced to parse technical plain text, which is particularly exclusionary for younger audiences who make up a large portion of the Minecraft community.#

Dependency information is frequently presented as technical metadata or documentation that many players cannot easily interpret. This creates barriers for younger or less technically experienced players who form a large portion of the Minecraft community.

[.line-through.red]#Furthermore, most mods do not display dependencies or incompatibilities clearly on their main mod pages.
Players need a way to understand these dependencies and conflicts without technical barriers, so they do not abandon using Minecraft mods due to confusion or frustration.#

Additionally, many mod pages do not clearly present dependency relationships or incompatibilities. Players often struggle to understand why a mod fails to work or which additional mods are required.


[.line-through.red]#There is, therefore, a need for a mod dependency visualization platform that allows users to explore mods and their dependencies in a clear and intuitive way.
Users need to see which mods depend on a selected mod and understand why each dependency is required.
The team will develop a comprehensive web-based platform designed to transform the way Minecraft players interact with mod configurations.
The approach is to move away from text-heavy logs and provide a Dynamic Dependency Graph.
This will make conflict detection visual and easy to manage for all users.#

A platform is therefore needed that allows players to visualize mod dependencies and understand relationships between mods more easily.


==== 1.4.1 Needs
Based on the current situation, several needs arise for players interacting with Minecraft mods:

* Players need clear and understandable information about mod dependencies so they can determine which additional components are required for a mod to function correctly.

* Players need accessible explanations of compatibility relationships between mods, including which versions, loaders, or libraries are required, without relying on technical logs or developer-oriented documentation.

* Players need an intuitive way to explore relationships between mods, including understanding which mods depend on others and how these dependencies affect the overall configuration.

* Players need support in identifying potential conflicts or missing components before launching the game, so they can resolve problems earlier in the installation process.

* Players also need a more approachable way to understand mod configurations, particularly for less technical players who may struggle with interpreting technical error messages or documentation.

==== 1.4.2 Ideas
To help satisfy these needs, the team proposes the development of a platform that assists players in exploring and understanding mod relationships.

* One idea is to provide visual representations of mod dependencies, allowing players to see how mods connect to each other rather than reading long lists of requirements. A graphical representation could make these relationships easier to understand.

* Another idea is to allow players to select a mod and view its dependency structure, including which additional mods are required and which other mods depend on it. This would help players better understand the role of each mod in their configuration.

* The system could also highlight missing dependencies or potential incompatibilities, helping players identify issues before attempting to run the game.

* Additionally, the platform could provide simplified explanations of dependency relationships, making technical information easier to interpret for a broader audience.

Overall, these ideas aim to improve how players interact with mod configurations by presenting dependency information in a clearer, more intuitive format.

=== 1.7 Concepts

* *Concept 1: Mod (Entity)* +
  A packaged software extension for Minecraft that adds or changes gameplay functionality and declares metadata such as version, supported Minecraft and Mod Loader versions, and dependency constraints.
* *Concept 2: Modpack (Entity)* +
  A selected set of Mods intended to run together under a specific Minecraft version and Mod Loader, whose correctness depends on satisfying all declared constraints.
* *Concept 3: Dependency (Relation)* +
  A directed requirement from one Mod to another Mod or library indicating that the target must be present and satisfy any stated Version Constraint for the source Mod to operate correctly within a Modpack.
* *Concept 4: Incompatibility (Relation)* +
  A declared constraint between two Mods or specific versions stating they must not coexist in the same Modpack due to functional or version conflicts.
* *Concept 5: Version Constraint (Attribute)* +
  A rule attached to a Dependency or Incompatibility declaration that specifies which versions of a target Mod are acceptable.
* *Concept 6: Dependency Resolution (Behavior)* +
  The process of determining whether a Modpack satisfies all required Dependencies, including transitive ones, and does not violate declared Incompatibilities or Version Constraints.

== 2.2 Narrative

Within the mod ecosystem of Minecraft, [.line-through.red]#users# players construct gameplay configurations by combining independently developed modifications, each of which declares structural constraints through metadata. A mod specifies its identity, supported versions, required dependencies, optional dependencies, and declared incompatibilities.When multiple mods are grouped into a modpack, they form a configuration whose validity depends on the collective satisfaction of all declared requirements and constraints. Adding or removing a mod triggers domain behaviours such as dependency expansion through transitive requirements, compatibility evaluation, and conflict detection.A configuration is considered structurally valid only if every required dependency is present in a compatible version and no declared incompatibilities exist among included mods. These relationships can be abstracted as a dependency network in which mods participate in directed requirement relations and constraint relations.[.line-through.red]#When these constraints are not satisfied, players may experience issues such as game crashes during startup, loading failures, missing content, or unstable gameplay behavior.In real scenarios, a player may attempt to launch the game after adding several mods and encounter immediate crashes, error messages, or incomplete loading of features. These situations often require trial-and-error actions, such as removing mods, changing versions, or searching for missing dependencies, highlighting the complexity of maintaining a valid mod configuration.# When dependencies are missing, players experience crashes on startup with error messages listing the absent mods. When version constraints are not satisfied, the game refuses to load and names each offending mod individually. When two mods declare mutual incompatibility as Canvas Renderer and Sodium do both may be present in the folder, the game loads, but one overrides or breaks the other, producing incorrect rendering, missing features, or runtime crashes with no clear indication of the conflict's source. In all cases, resolving the problem requires players to manually trace relationships across multiple mod pages, changelogs, and community forums, with no unified view of how their mods interact.

=== 2.4 Terminology

This section defines the core terms that will be used throughout the project.
These definitions apply to both the backend dependency logic and the frontend graph interface.

A key distinction in this terminology is the difference between domain concepts and system concepts.
Domain concepts refer to phenomena that exist in the Minecraft modding ecosystem independently of the application, such as Mod, Dependency, Incompatibility, Mod pack, and Version Constraint.
System concepts are introduced by the design and implementation of the project itself, such as Mod Instance, Dependency Graph, Node, Edge, UploadQueue, and ValidationFeedback.

This distinction helps avoid mixing properties of the observed domain with structures that only exist inside the system.
As a result, the terminology supports clearer requirements, more consistent [.line-through.red]#msodeling# modeling, and a better separation between domain understanding and implementation design.

Two additional distinctions are worth clarifying upfront. First, Events and Actions are not the same thing: an Event is instantaneous and marks a transition between states (e.g., a conflict is detected), while an Action has duration and describes something the system or user does over time (e.g., resolving dependencies). Second, a State is a condition that persists until something changes it — a Modpack being in conflict is a State, not an Event. These distinctions affected several classifications below.

Additionally, the Minecraft base game is treated as a mod-like entity in this model. It has an identifier (`minecraft`) and a version (e.g., `1.21.1`), and mods declare dependencies toward it the same way they do toward other mods. This lets the system handle Minecraft version constraints the same way it handles any other dependency constraint.

Each term includes:

* [.line-through.red]#Classification (Entity, Relation, Attribute, Event, Behavior)#
* Classification (Entity, Relation, Attribute, Event, Behavior, State)
* Phase Introduced (Domain = Minecraft ecosystem concept, System = project-specific concept)
* Definition (Implementation-independent)

. Mod (Modification) +
  Classification: Entity +
  Phase Introduced: Domain +
  Definition: A mod (modification) is a packaged software extension that alters or adds functionality to Minecraft. A mod may introduce new content, mechanics, or systems and may declare dependencies or incompatibilities with other mods.

. [.line-through.red]#Dependency# +
  [.line-through.red]#Classification: Relation# +
  [.line-through.red]#Phase Introduced: Domain# +
  [.line-through.red]#Definition: A dependency is a directed relationship in which one mod requires another mod in order to function correctly. If the required mod is missing or does not satisfy the stated version constraint, the dependent mod cannot operate as intended.# +
+
Dependency +
Classification: Relation +
Phase Introduced: Domain +
Definition: A directed relation from a source Mod to a target Mod, stating that the source requires the target to be present and satisfy any associated Version Constraint to function correctly. A dependency cannot exist without both Mods — it is always between two specific participants. +
+
A dependency is classified as a Relation rather than an Entity because it has no existence on its own; it only makes sense in the context of two Mods. It is not a Behavior either, since it does not describe a process — it describes a structural constraint that exists regardless of whether resolution has been run. When a dependency carries a specific version requirement (e.g., `fabric-api >= 0.90.0`), it is a versioned dependency. When no version is specified, any version of the target Mod is acceptable. +
+
Example: Sodium declares a dependency on Fabric API. When a player updates Sodium, its version constraint on Fabric API may tighten, causing a previously working Modpack to break if the installed Fabric API no longer qualifies.

. Optional Dependency +
  Classification: Relation +
  Phase Introduced: Domain +
  Definition: A relationship where a mod can integrate with another mod if it is present, but does not require it for core functionality. Like Dependency, it connects two specific Mods and cannot exist independently, so it is classified as a Relation for the same reasons.

. Version +
  Classification: Attribute +
  Phase Introduced: Domain +
  Definition: A structured identifier assigned to a specific release of a mod. Versions are used to distinguish releases and evaluate compatibility between mods.

. [.line-through.red]#Version Constraint# +
  [.line-through.red]#Classification: Attribute# +
  [.line-through.red]#Phase Introduced: Domain# +
  [.line-through.red]#Definition: A rule that specifies which versions of a mod are acceptable within a dependency relationship. A version constraint defines a valid range or condition that must be satisfied.# +
+
Version Constraint +
Classification: Attribute +
Phase Introduced: Domain +
Definition: A rule attached to a Dependency relation that specifies which versions of the target Mod are acceptable (e.g., `>= 0.90.0`, `^1.4.0`). The constraint must be satisfied for the dependency to be considered resolved. +
+
Version Constraint is an Attribute because it is a property of a Dependency — it does not exist on its own. When two dependencies in the same Modpack impose conflicting constraints on the same target Mod, the Modpack enters a Conflict state that cannot be fixed without changing one of the source Mods. +
+
Example: Indium 1.0.21 requires `fabric-api <= 0.88.1`, while Geckolib 4 requires `fabric-api >= 0.90.0`. No single version of Fabric API satisfies both, so the Modpack is in conflict.

Minecraft Version +
Classification: Entity +
Phase Introduced: Domain +
Definition: A specific release of Minecraft and its updates. It defines the base environment and APIs that mods and mod loaders target, and is central to compatibility constraints in the dependency graph. +
+
Within the dependency model, the Minecraft base game behaves as a mod-like entity with identifier `minecraft` and a version string (e.g., `1.21.1`). Mods may declare a dependency on `minecraft` with a Version Constraint, resolved the same way as any other dependency. This allows the system to catch cases where a player runs a Modpack on a Minecraft version the mods do not support. +

Example: A player moves their Modpack from Minecraft 1.20.1 to 1.21.1. Several mods declare `minecraft ~1.20` as a dependency. The system flags each of them as having an unresolved dependency against the new version.

  [.line-through.red]#Classification: Behavior# +
  [.line-through.red]#Phase Introduced: System# +

Resolution +
Classification: Behavior +
Phase Introduced: System +
Definition: The process of evaluating whether all declared Dependency relations within a Modpack are satisfied according to their Version Constraints and compatibility rules. It begins when a Modpack is submitted for analysis and ends when a Resolution Result is produced. +
+
Resolution is a Behavior because it takes time — it is something the system does, not a moment in time. The moment it finishes is an Event (see section 2.7). What might look like separate processes — checking version rules, detecting conflicts, expanding transitive dependencies — are all phases of Resolution, not independent actions.

. [.line-through.red]#Conflict# +
  [.line-through.red]#Classification: Event# +
  [.line-through.red]#Phase Introduced: System# +
  [.line-through.red]#Definition: A condition that occurs when dependency requirements or version constraints cannot be satisfied simultaneously. A conflict prevents successful resolution of the mod set.# +
+
Conflict +
Classification: [.line-through.red]#Event# State +
Phase Introduced: System +
Definition: A condition in which two or more Mods in a Modpack cannot coexist due to a declared Incompatibility or an unsatisfiable Version Constraint. The Modpack remains in this state until the problem is corrected. +
+
Conflict was reclassified from Event to State because it is not instantaneous — it is a condition the Modpack sits in over time. The moment the conflict is first detected is the Event (see "Incompatibility Detected" in section 2.7). +
+
Example: A player adds Canvas Renderer to a Modpack that already has Sodium. Since Canvas Renderer declares Sodium as incompatible, the Modpack enters a Conflict state and stays there until one of them is removed.

. [.line-through.red]#Compatibility# +
  [.line-through.red]#Classification: Attribute# +
  [.line-through.red]#Phase Introduced: Domain# +
  [.line-through.red]#Definition: The state in which two or more mods, versions, or loaders can operate together without violating declared dependencies or constraints.# +
+
Compatibility +
Classification: [.line-through.red]#Attribute# State +
Phase Introduced: Domain +
Definition: The condition in which two or more Mods, versions, or loaders can coexist in a Modpack without violating any declared Dependency relations, Incompatibility relations, or Version Constraints. +
+
Compatibility was reclassified from Attribute to State because it describes the overall condition of a configuration at a point in time, not a property of any single Mod. It can change — a Modpack that is compatible today may become incompatible after a mod update. +
+
Example: Sodium and Fabric API are compatible as long as the installed Fabric API version satisfies Sodium's constraint. If the player updates Sodium and the new version requires a higher Fabric API version than what is installed, the Modpack moves from a compatible to an incompatible state.

. Installation State +
  Classification: Attribute +
  Phase Introduced: System +
  Definition: The current status of a mod instance within the managed environment, such as installed, missing, unresolved, or conflicting.

  [.line-through.red]#Classification: Relation# +
  [.line-through.red]#Phase Introduced: System# +

Transitive Dependency +
Classification: Relation +
Phase Introduced: System +
Definition: A dependency that arises indirectly through a chain of other dependencies. If Mod A depends on Mod B, and Mod B depends on Mod C, then Mod A has a transitive dependency on Mod C. +
+
Like Dependency, this is classified as a Relation. The difference is that transitive dependencies are not declared explicitly in any Mod's metadata — they emerge from following the chain of declared relations during Resolution. +
+
Example: A player installs Autochef's Delight, which depends on Farmer's Delight, which depends on Fabric API. The Modpack ends up with a transitive dependency on Fabric API even though Autochef's Delight never mentions it.

. Profile +
  Classification: Entity +
  Phase Introduced: System +
  Definition: A named container that stores a user's mod instances and their installation statuses.

. Incompatibility +
  Classification: Relation +
  Phase Introduced: Domain +
  Definition: A declared constraint between two Mods stating they must not coexist in the same Modpack. +
+
Incompatibility is a Relation because it connects two specific Mods and has no meaning without both. It is worth distinguishing from Conflict: Incompatibility is the rule declared in a Mod's metadata, while Conflict is the State that results when that rule is violated in a Modpack. +
+
Example: Canvas Renderer declares an Incompatibility with Sodium. If a player includes both in a Modpack, the result is a Conflict state.

. Versioned Dependency +
  Classification: Relation +
  Phase Introduced: Domain +
  Definition: A Dependency relation that carries an explicit Version Constraint on the target Mod. The source Mod functions correctly only if the target is present at a version within the declared range. +
+
Many of the dependency problems seen in the Minecraft modding community come specifically from versioned dependencies where the acceptable version range is narrow or clashes with the range required by another mod in the same Modpack. +
+
Example: Indium 1.0.21 requires `fabric-api <= 0.88.1` and Geckolib 4 requires `fabric-api >= 0.90.0`. Since no version of Fabric API satisfies both at once, the Modpack is in conflict.

=== 2.8 Software Architecture Design

The system follows a layered client - server architecture composed of four main components:

* Frontend (Presentation Layer): Handles mod upload, user interaction, search functionality, and visualization of the dependency graph.
  It renders mods as nodes and dependencies as directed edges, and provides visual indicators for missing, satisfied, or incompatible dependencies.
* Backend API Layer: Receives uploaded metadata, validates input, coordinates processing, and returns structured JSON results to the frontend.
  It also manages request handling, logging, and retry mechanisms for external API calls.
* Dependency Resolution Engine (Core Layer): Parses mod metadata, normalizes dependency information, resolves required and optional dependencies, detects incompatibilities and circular dependencies, validates version constraints, expands transitive dependencies, and constructs the internal dependency graph model.
* Data & Integration Layer: Communicates with external mod repositories such as Modrinth and CurseForge.
  It handles metadata retrieval, caching of previously accessed data, and graceful error handling when external services are unavailable.

==== Team A Dependency Graph Subsystem

Team A is responsible for the dependency graph subsystem, which models mods as nodes and dependency relationships as directed edges.

Its main role is to build and maintain the internal graph structure used for dependency analysis and later visualization.
This includes preserving node uniqueness, representing dependency relationships consistently, and producing structured graph data for the frontend.

This subsystem is kept separate from upload handling, external API communication, and frontend rendering.
Such separation improves modularity, maintainability, and testability by isolating graph logic from other system concerns.

Within the overall architecture, the Team A graph subsystem connects dependency resolution results with the frontend visualization layer.

==== Team B External API integration and Dependency Resolution Subsystem

Team B is responsible for the external API integration and dependency resolution subsystem. Its main role is to communicate with the Modrinth API to retrieve mod metadata, resolve declared dependencies, and detect missing or circular dependencies given a modpack configuration.

The subsystem recieves a Modrinth project slug as its entry point and executes a sequence of API calls to retrieve the project ID, available versions, and declared dependencies. For each missing required dependency, the subsystem quieres the Modrinth API filtered by mod loader and Minecraft version to find a compatible version and extract its downloaded URL.

Results are packaged into a structured ValidationResponse containing missing dependencies, circular dependency chains, and resolved dependency entries with download URL's. 

This subsystem is kept separate from the graph and frontend visualization rendering, communicating with the rest of the system through well defined DTO's. Such seperation impoves modularity and makes the external API behaviour testable. 

=== Team C Frontend Web App and Visualization: 

  Team C is responsible for the frontend web application layer. This includes accepting mod/modpack uploads in .zip format from players through a file picker interface and transmitting the uploaded uploaded file to the backend via a POST requst to the /api/modpack/zip endpoint. 
  
  Upon recieving the processed dependency data from the backend, the frontend renders the dependency graph and displays resolved and unresolved dependencies to the player. 
  
  Team C also implements client-side dependency resolution logic, including functions for retrieving direct dependencies and reverse dependents from the graph data structure, and maintains the graph state used by the visualization layer. The subsystem validates uploaded file formats before transmission and provides real-time status feedback to the player during the upload process.
